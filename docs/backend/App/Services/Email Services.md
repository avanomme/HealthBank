# `markdown/email.md` — Email Delivery (SMTP + Templates)

This project currently contains **two parallel email implementations**:

1. **`backend/app/services/email_service.py`**  
   - A standalone async-friendly SMTP sender (`EmailService.send_email()` uses `run_in_executor`).
   - Config comes from `backend/app/core/config.py` via `get_smtp_config()`.

2. **`backend/app/services/email/` package**  
   - A provider-based abstraction (`EmailProvider`, `GmailProvider`) plus `EmailService` wrapper and templates.
   - Config comes from `backend/app/services/email/config.py` (env vars `GMAIL_USER`, `GMAIL_APP_PASSWORD`, plus application URLs).
   - Templates live in `backend/app/services/email/templates.py`.

They overlap in capability, and long-term you should pick one approach and delete/merge the other to avoid drift.

---

## A) SMTP Email Service (current async sender)

### File
- `backend/app/services/email_service.py`

### What it does
- Creates multipart emails with:
  - a **plain-text fallback** (generated by stripping HTML if not provided)
  - the **HTML version**
- Sends via SMTP over **STARTTLS**
- Returns an `EmailResult { success, message, recipient }`
- Provides:
  - `await send_email(...)` (async)
  - `send_email_sync(...)` (blocking)

### Dependencies
- `backend/app/core/config.py`
  - `get_smtp_config()` returns an `SMTPConfig` dataclass.

### SMTPConfig inputs (env vars)
Defined in `backend/app/core/config.py`:

- `SMTP_HOST` (default `smtp.gmail.com`)
- `SMTP_PORT` (default `587`)
- `SMTP_USER` (smtp username / from address)
- `SMTP_PASSWORD` (smtp password / app password)
- `SMTP_FROM_NAME` (default `HealthBank`)
- `SMTP_TIMEOUT` (default `30`)

### Behavior notes
- If credentials are missing (`config.is_configured == False`), it **does not send**, returns a failure result.
- STARTTLS + login + sendmail using `self.config.user` as the envelope sender.

## B) Provider-based Email Module (templates + Gmail provider)

### Files
- `backend/app/services/email/base.py`
- `backend/app/services/email/gmail.py`
- `backend/app/services/email/service.py`
- `backend/app/services/email/config.py`
- `backend/app/services/email/templates.py`

### What it does
- Defines a provider interface (`EmailProvider`) and a Gmail SMTP implementation (`GmailProvider`).
- Wraps provider usage in a higher-level `EmailService` with template helpers like:
  - `send_account_created_email(...)`
  - `send_password_reset_email(...)`
  - `send_password_changed_email(...)`
  - `send_2fa_setup_email(...)`
- Templates return `(subject, body_text, body_html)`.

### Config (env vars)
Defined in `backend/app/services/email/config.py`:

- `GMAIL_USER`
- `GMAIL_APP_PASSWORD`

Plus application URLs:

- `APP_BASE_URL` (default `http://localhost:3000`)
- derived:
  - `LOGIN_URL`
  - `PASSWORD_RESET_URL`
  - `TWO_FACTOR_SETUP_URL`

Validation:
- `validate_email_config()` raises if required values are missing.

## Current Usage in Your API

Your `admin.py` uses the **SMTP Email Service** approach:

- `from ...services import get_email_service, password_reset_email, password_reset_plain`

and then:

- `email_service = get_email_service()`
- `await email_service.send_email(...)`

So operationally, `backend/app/services/email_service.py` is the implementation being used for admin-triggered reset emails.

---

## Recommendation (to reduce duplication)

### Option 1 (simplest): Standardize on `services/email_service.py`
Keep:
- `backend/app/services/email_service.py`
- `backend/app/core/config.py`
- whichever template functions you already call (`password_reset_email`, `password_reset_plain`)

Remove or archive:
- the provider-based `backend/app/services/email/` module

### Option 2 (more extensible): Standardize on `services/email/`
Keep:
- provider-based module + templates

Refactor:
- change `admin.py` to call the provider-based `EmailService` functions
- unify env vars (either rename `SMTP_*` to match `GMAIL_*` or vice versa)

Remove:
- `backend/app/services/email_service.py`

Either path is fine—just pick one so you don’t maintain two stacks.

## Quick “gotchas” checklist

- Ensure your SMTP credentials are **app passwords** if using Gmail.
- If deployed behind a proxy, verify outbound SMTP is allowed.
- Plain-text fallback generation in `email_service.py` is intentionally simple (regex HTML strip) and may produce “run-on” text; passing an explicit `plain_content` is better when possible.
- Consider setting `msg["Reply-To"]` if you want responses routed somewhere (currently templates say “do not reply”).
